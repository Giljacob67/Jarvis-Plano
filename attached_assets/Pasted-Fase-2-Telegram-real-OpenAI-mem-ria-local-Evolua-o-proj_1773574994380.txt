Fase 2 — Telegram real + OpenAI + memória local

Evolua o projeto existente de forma incremental. Não recrie o scaffold. Não adicione Node, frontend, package.json ou outra stack. Preserve FastAPI, a estrutura modular atual e os endpoints da fase 1.

Objetivo
Transformar o “Jarvis Pessoal” em um bot funcional no Telegram para 1 usuário, texto-first, com memória local em SQLite e integração real com OpenAI, pronto para depois plugar Google Workspace.

Escopo desta fase
- integração real de entrada e saída com Telegram para mensagens de texto
- OpenAIService funcional usando Responses API
- memória local + histórico de conversa
- segurança básica + idempotência
- preparação para voz sem transcrição ainda
- sem Google real nesta fase

Restrições
- não quebrar a fase 1
- sem Docker
- sem filas externas
- sem dependências desnecessárias
- tudo em pt-BR nas respostas do assistente
- se OPENAI_API_KEY não estiver configurada, responder de forma amigável sem crashar

3) Telegram real
Evoluir `TelegramService` para:
- validar o header `X-Telegram-Bot-Api-Secret-Token`
- ignorar com HTTP 200 qualquer mensagem cujo user_id seja diferente de `TELEGRAM_ALLOWED_USER_ID`
- tratar `update_id` como idempotency key e ignorar updates duplicados com HTTP 200
- enviar respostas reais usando a Bot API `sendMessage`
- não logar tokens nem segredos

Implementar estes comportamentos:
- `/start` -> mensagem curta apresentando o assistente
- `/help` -> listar comandos disponíveis
- `/myday` -> responder usando a lógica mock atual do endpoint `/me/day`
- `/remember <texto>` -> salvar uma memória simples
- `/memories` -> listar as últimas memórias ativas
- texto livre -> chamar OpenAIService e responder no Telegram

4) OpenAIService funcional
Implementar `OpenAIService` com o client oficial da OpenAI usando a Responses API.

Criar um método principal parecido com:
`generate_reply(user_id: str, user_text: str, recent_messages: list, memories: list) -> str`

Requisitos do OpenAIService:
- usar `OPENAI_MODEL` do ambiente
- criar um system prompt fixo em pt-BR focado em produtividade pessoal
- usar poucas tools internas por function calling:
  - `get_my_day()`
  - `save_memory(note, category)`
  - `list_memories(limit)`
- `get_my_day()` deve reutilizar a lógica existente do `/me/day`
- limitar o contexto enviado ao modelo às mensagens recentes + memórias relevantes
- se o usuário pedir ação sensível nesta fase (ex.: enviar e-mail, apagar algo, cancelar evento, editar agenda real), não executar nada; responder que essa ação será habilitada numa fase com aprovação explícita e registrar isso no `ActionLog`
- se a OpenAI estiver sem configuração, retornar mensagem amigável tipo “a integração com OpenAI ainda não foi configurada”

5) Memória
Implementar memória simples e útil:
- comando `/remember <texto>` salva memória com category=`general`
- tool `save_memory` também salva memória
- tool `list_memories` retorna memórias recentes
- criar helpers para recuperar memórias relevantes por texto simples
- sem vetor database nesta fase
- sem embeddings nesta fase
